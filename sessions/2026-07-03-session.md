## What We Found

- The database committed to the repo (`data/collaboration_network.db`, 27,442 artists) was never a clean depth-2 build — it was a partially-completed depth-3 build made with **pre-pagination-fix** code, committed hours before that fix landed. Only ~846 of its artists were genuinely fully crawled (the rest are leaf nodes discovered but never processed).
- Three separate "hang" incidents this session, each initially misdiagnosed as network/threading/SQLite bugs, but the **real root cause of the first two was code bugs, and the third was Spotify's actual rate-limit system**:
  1. **Missing request timeout** — `requests.post`/`requests.get` in `src/data_fetcher.py` had no `timeout=`, so a stalled connection during a network disruption blocked a worker thread forever.
  2. **`ThreadPoolExecutor` can't be interrupted** — even with a request timeout, Python threads can't be force-killed, and `ThreadPoolExecutor`'s own `with`-block shutdown waits for every submitted task regardless.
  3. **The actual root cause of the final ~2-hour "stall"**: a live, direct request to Spotify's API returned HTTP 429 with `Retry-After: 35268` (~9.8 hours) — this app's Spotify credentials were genuinely rate-limited from the cumulative volume of the day's repeated crawl attempts and diagnostic tests. `_make_request`'s retry loop was doing a blind `time.sleep(retry_after)`, which is indistinguishable from a hung process when the value is that large.
  4. Ruled out along the way (real, methodical elimination, not guesses): SQLite locking (DB opened instantly, integrity check passed), general Python socket-timeout mechanics (verified working against a synthetic black-hole IP), IPv6 path issues (forced IPv4-only DNS resolution as a hardening measure — did not fix the actual stall, confirming it wasn't the cause), single-threaded vs. concurrent requests.
  5. Confirmed via official Spotify docs: Spotify does not publish an exact numeric rate limit — only that it's calculated on a rolling 30-second window. Our own `RateLimiter(max_requests=50, window_seconds=30)` was always a guess.
  6. **As of this morning (2026-07-03), the rate limit has cleared** — a fresh direct request against the existing Spotify credentials returned 200, not 429.

## What We Fixed or Built

- `src/data_fetcher.py`:
  - Added `timeout=30` to both `requests.post` (auth) and `requests.get` (API calls).
  - Added `socket.getaddrinfo` monkeypatch forcing IPv4-only DNS resolution (hardening measure; didn't turn out to be the root cause, but kept as a reasonable safeguard).
  - Added `RateLimitPenaltyError` exception, raised by `_make_request` when `Retry-After` exceeds a 60-second inline-retry cap, instead of blindly sleeping for hours.
  - Lowered `_rate_limiter`'s default from `max_requests=50` to `max_requests=20` per 30s window, as a safety margin (Spotify's real threshold is undisclosed).
- `src/build_network_sqlite.py`:
  - Rewrote `build_network()`'s batch-collection loop to use `concurrent.futures.wait(..., timeout=90, return_when=FIRST_COMPLETED)`, giving up on stuck futures after 3 consecutive stalls (~4.5 min) and calling `executor.shutdown(wait=False, cancel_futures=True)` instead of the old `as_completed()` + implicit `with`-block shutdown (which could never be interrupted).
  - Added a circuit breaker: `process_single_artist` and `build_network`'s collection loop both now explicitly re-raise `RateLimitPenaltyError` past their broad `except Exception` handlers (this was caught as a P0 feasibility-review finding — as originally drafted, the fix would have been silently swallowed by exactly those two catch-alls, recreating the same failure mode). **Verified end-to-end via simulation**: a monkeypatched `RateLimitPenaltyError` correctly propagates all the way out of `build_network()`, while an ordinary unrelated exception is still correctly swallowed per-artist (no regression).
  - Added proactive rate-limit budget heartbeat logging (every ~30s wall-clock, not just every 10 completed artists) so liveness is visible during a slow-but-healthy patch.
  - Added `--resume` and `--fresh` non-interactive CLI flags; fixed a real bug where the "clear existing data" step wasn't actually gated on the user's y/n response.
  - Added `main()`-level `sys.exit(1)` on failure paths instead of silent `return` (previously exited 0 even on real errors).
- `src/database.py`: added a `crawled INTEGER DEFAULT 0` column to `artists` (with an `ALTER TABLE` migration guard), plus `mark_artist_crawled()` / `is_artist_crawled()` / `get_artist_neighbors()` methods — this is the resumability mechanism. Note: the first design (inferring "crawled" from edge presence in the undirected `collaborations` table) was also caught as a P0 feasibility-review finding and replaced with this explicit column.
- `app.py`: replaced the misleading "Artist not found" message with an honest "isn't in our network yet" message (merged, PR #18).
- `STRATEGY.md`, `docs/ROADMAP.md`: written/updated this session.

## Decisions Made

- **"Rabbit Hole" is the umbrella brand/identity** for this whole project going forward; the Kendrick collaboration graph is Track 1, not the whole product.
- **Clean depth-3 rebuild from scratch, not incremental reuse** of the questionable existing data — accuracy over the ~30-35% time savings incremental reuse would have offered.
- **No live Spotify existence-check or on-demand network expansion** for the "not in network" case — kept as a pure messaging fix, deferred richer fallback behavior to later.
- **Rebuild runs in the user's own terminal** (`nohup` + `disown`), decoupled from any AI session's lifecycle, after the original overnight attempt (running via the AI session's sandbox) got disrupted by a Claude account/profile switch.
- **20 req/30s is a safety-margin choice, not a provably safe number** — Spotify doesn't publish an exact threshold, so this trades crawl speed for a larger margin, informed directly by today's real ~9.8 hour penalty at the old 50 req/30s setting.
- **A large rate-limit penalty aborts the entire crawl immediately**, not just the current artist or level — continuing to hit the API under an active penalty risks extending it.
- **(This session, in progress) Do NOT register a new Spotify developer app.** Earlier in the previous session the user was leaning toward a fresh app for a clean rate-limit bucket, but since 10+ hours have now passed and the existing credentials are confirmed working again (200 response, not 429), the plan is to reuse the existing key rather than add the complexity of a new one.
